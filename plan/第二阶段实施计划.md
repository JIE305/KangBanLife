# 第二阶段实施计划

## 目标

将前端 5 个仍使用 Mock 数据的模块逐步迁移到真实后端 API，同时补齐后端对应的数据库表和 API 端点。

## 当前差距

| 模块 | 前端 | 后端 | 数据库表 |
|------|------|------|---------|
| 日常管理 Health | Mock | 无 API | 无（需建 user_health_data, user_trend_data） |
| 配餐 Meals | Mock | 无 API | 无（需建 meal_recommendations） |
| 运动 Sports | Mock | 无 API | 无（需建 sport_courses, medals, user_medals, sport_matches） |
| 资源 Resources（完整版） | Mock | 部分 API 存在但表未建 | 无（需建 first_aid_resources, medicines, medical_facilities） |
| 心情 Mood | Mock | 无 API | 无（需建 mood_records） |

> 注：Resources 的 `api/resources.py` 已写了 4 个端点的查询代码，但 `schema_lite.sql` 未建对应 3 张表，SQL 查询会报错。

---

## 实施优先级

### 优先级 1：资源模块 Resources（工作量最小）

**原因**：后端 API 代码已写完（`api/resources.py`），只需建表 + 种子数据 + 前端对接。

**后端**：
- 在 `schema_lite.sql` 中添加 `first_aid_resources`、`medicines`、`medical_facilities` 建表 SQL
- 在 `seed_data.py` 中添加资源种子数据
- 重新运行 `seed_data.py`

**前端（Resources.vue）**：
- 创建 `src/api/resources.js`（4 个端点）
- 替换硬编码的急救知识、药品清单、校医室信息为 API 调用
- 绑定搜索框的 `keyword` 筛选逻辑
- 周边医疗设施从硬编码切换到 API 搜索

**预估工作量**：2-3 小时

### 优先级 2：日常管理 Health

**原因**：前端 UI 已完整（饮水/清单/BMI/餐食/趋势图），数据模型清晰。

**后端**：
- 在 `schema_lite.sql` 中建 `user_health_data`、`user_trend_data` 表
- 新建 `api/health.py` 蓝图（注册在 `/api/v1/health`）
  - `GET /health/daily` — 当日健康数据
  - `POST /health/daily` — 提交/更新饮水/步数等
  - `GET /health/trend?days=7` — 趋势数据
  - `POST /health/bmi` — 记录 BMI
- 在 `app.py` 中注册 health 蓝图

**前端（Health.vue）**：
- 创建 `src/api/health.js`
- 饮水记录：`POST /health/daily` 持久化
- BMI 计算：`POST /health/bmi` 保存历史
- 趋势图：`GET /health/trend` 获取真实数据
- 清单状态：存入 `user_health_data`

**预估工作量**：4-5 小时

### 优先级 3：心情模块 Mood

**原因**：前端已有心情选择 UI，功能简单，表结构单一。

**后端**：
- 在 `schema_lite.sql` 中建 `mood_records` 表
- 新建 `api/mood.py` 蓝图
  - `POST /mood` — 提交今日心情
  - `GET /mood?days=7` — 近期心情记录

**前端（Mental.vue）**：
- 心情选择后调用 `POST /mood` 保存
- 按日期展示心情历史（可选图表）

**预估工作量**：2-3 小时

### 优先级 4：配餐模块 Meals

**原因**：数据需要人工整理（食堂真实菜品），可用种子数据先行。

**后端**：
- 在 `schema_lite.sql` 中建 `meal_recommendations` 表
- 新建 `api/meals.py` 蓝图
  - `GET /meals` — 按 meal_type（breakfast/lunch/dinner）筛选
  - `GET /meals/recommended` — 今日推荐

**前端（Health.vue 餐食部分或独立 Meal.vue）**：
- 替换硬编码的 `mealOptions` 为 API 数据
- 按餐次筛选联动

**预估工作量**：3-4 小时

### 优先级 5：运动模块 Sports（工作量最大）

**原因**：涉及 4 张表、课程库、勋章系统、约球匹配，复杂度最高。

**后端**：
- 在 `schema_lite.sql` 中建 `sport_courses`、`medals`、`user_medals`、`sport_matches` 表
- 新建 `api/sports.py` 蓝图
  - `GET /sports/courses` — 课程列表（按类型筛选）
  - `POST /sports/matches` — 发布约球
  - `GET /sports/matches` — 约球列表
  - `GET /sports/medals` — 用户勋章
  - `POST /sports/checkin` — 运动打卡（触发勋章检测）

**前端**：
- 创建 `src/api/sports.js`
- 课程库从硬编码切换到 API
- 课程标签（全部/拉伸/力量）绑定筛选
- 约球广场 CRUD
- 勋章系统动态展示

**预估工作量**：6-8 小时

## 实施顺序建议

```
第一步：资源 Resources  ──→ 立即见效，快速验证对接流程
第二步：心情 Mood        ──→ 简单独立，快速积累经验
第三步：日常管理 Health   ──→ 核心功能，用户感知强
第四步：配餐 Meals       ──→ 依赖数据整理
第五步：运动 Sports      ──→ 最复杂，最后攻克
```

## 每模块的通用实施步骤

1. **后端建表** — 在 `schema_lite.sql` 中添加建表 DDL
2. **后端蓝图** — 在 `api/` 下新建 `{module}.py`
3. **注册蓝图** — 在 `app.py` 中添加注册
4. **种子数据** — 在 `seed_data.py` 中添加测试数据
5. **前端 API 模块** — 在 `src/api/` 下新建 `{module}.js`
6. **前端视图改造** — 修改 `{Module}.vue` 替换 Mock 数据
7. **手动测试** — 用 curl 或浏览器验证 CRUD
8. **更新文档** — 在 CHANGELOG.md 中记录，同步更新受影响的 plan/ 文档

## 依赖关系

```
Resources ← 无依赖，可最先开始
Mood     ← 依赖用户认证（已有）
Health   ← 依赖用户认证（已有）
Meals    ← 无依赖（公开数据）
Sports   ← 依赖用户认证 + 勋章逻辑依赖打卡数据
```

Resources 和 Meals 可并行开发（都是公开数据），其余三个按优先级串行。
